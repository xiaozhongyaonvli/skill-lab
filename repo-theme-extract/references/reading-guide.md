# 阅读指南.md · 结构模板与写作要点

`阅读指南.md` 是案例产物里**质量最重要的一份文件**。代码精简得再漂亮,如果没有这份"引路文件",读者上来还是会迷路。

## 标准骨架(章节顺序)

```
# 阅读指南:<案例名 + 一个能立论的小标题>

> (引语段:配套博客/讲义的指向 + 一句"本目录已为 X 主题精简"。可选。)

## 这堆文件想让你看到什么
   ── 一句话立论 + ASCII 大图(必须有图)

## 建议阅读顺序
   ── 5 站左右。每站的写法见下。

## 文件角色对照表
   ── 必备表格 1

## 和 <对照案例> 的差异表
   ── 必备表格 2(最重要,体现案例的洞察)

## (可选)和 <教科书例子> 的一一对应
   ── 表格 3,把代码映射到读者已知的概念上

## 读完之后的一个建议练习
   ── "新增 X 要改哪几个文件?分别改什么?"

## (可选)还想深入的话
   ── 指向原仓库的进一步阅读路径
```

5 个章节是骨架,后两个章节可选。**前 5 个章节缺一不可。**

---

## 各章节写法

### "这堆文件想让你看到什么"

两个组成:

1. **一句话立论**——这是整篇案例最重要的一句话,它把案例从"读懂代码"提升到"看懂模式"。例:
   > 策略模式在工程里【不总是函数指针表】。在追求内存与性能的场景下,它退化成了"一个 int 枚举 + if/else 分发"。

2. **ASCII 大图**——把所有文件按层级摆好,显示数据/控制流。例:
   ```
   ┌──────────────┐
   │  用户态       │   config 文件 / CONFIG SET 命令
   └──────┬───────┘
          │  (字符串名)
          ▼
   ┌──────────────┐
   │  注册表      │   string → enum 的映射(file A)
   └──────┬───────┘
          │  (解析后)
          ▼
   ┌──────────────┐
   │  全局状态    │   持有当前策略 id(file B)
   └──────┬───────┘
          │  (被读)
          ▼
   ┌──────────────┐
   │  主角        │   核心分发逻辑(file C)
   └──────────────┘
   ```

   **层次按"从外往里"或"从抽象到具体"摆**。不要按字母顺序、不要按目录顺序。

### "建议阅读顺序"——五站模板

每一站固定的小结构:

```
### 第 N 站:<这一站讲什么>

**文件:`<相对路径>`**(精简版,约 <X> 行)

(指向具体段落:"只看 XXX 这一段,在原 line A~B 附近")

读的时候注意:

1. <要点 1>
2. <要点 2>
3. <要点 3>

**读完应该明白**:<一句话凝练>。
```

**关键约定**:
- 顺序按"从抽象到具体":身份(常量定义)→ 选择(注册表)→ 使用(主角)→ 细节(辅助函数)→ 副作用(对存储/外部命令的反向影响)
- 每一站都用 **"读完应该明白"** 做收束——这是质量验收点,读者读完读不出这一句就说明这一站没讲清。

### "文件角色对照表"

```markdown
| 文件 | 在 <主题> 里的角色 | 大小 |
|---|---|---|
| `path/a.c` | <一句话角色> | ~N 行 |
| `path/b.h` | <一句话角色> | ~N 行 |
```

**关键**:第二列必须是**一句话**,不是几行说明。这一句话就是阶段 3 精简时"删够了"的硬验收。

### "和 <对照案例> 的差异表"

**这张表是案例的灵魂**。写法:

```markdown
| 维度 | <标准/对照实现> | <本案例> |
|---|---|---|
| 策略对象的形态 | 函数指针表 | 1 个 int 编码 |
| 客户端是否出现具体策略名? | ❌ 不出现 | ✅ 出现 |
| 新增策略是否需要改客户端? | ❌ 不需要 | ✅ 必须改 |
| 注册方式 | 运行时动态 | 编译期静态 |
| ... | ... | ... |
```

**维度选择**:5~10 个维度,覆盖**策略对象形态、注册方式、客户端耦合、扩展成本、运行时切换能力、对存储的影响、设计目标**。

**为什么这张表是案例的灵魂**:读者读完代码会觉得"哦原来 Redis 这样做",但不会主动去对比"标准做法和这样有什么差异"。这张表把对比强制摆出来,读者一眼看到"差异在哪、为什么"——这就是把案例从"读懂"提升到"看懂模式"的关键。

**结尾**带一段评论,例:
> 这两种长相是同一个模式的两端。理解这种差异比背"策略模式 UML 图"重要得多——它告诉你:模式是用来解决问题的,不是用来证明纯粹的。

### "和 <教科书例子> 的一一对应"(可选)

如果案例对应一个教科书例子(例:HF 设计模式的鸭子例子),用一张映射表把代码概念映射回去:

```markdown
| 教科书例子 | <本案例> | 在哪个文件 |
|---|---|---|
| `interface FlyBehavior` | (没有显式接口,用枚举代替) | <文件> |
| `class FlyWithWings` | `MAXMEMORY_ALLKEYS_LRU` 这一类宏 | <文件> |
| `duck.performFly()` | `if (policy & FLAG_LRU) { ... }` | <文件> |
```

最后一行经常出现"(没有)"——这是案例**比教科书例子多出来的工程考虑**,要点出来。

### "读完之后的一个建议练习"

固定形式:
```
合上这堆文件,尝试用 200 字回答:

> "如果我要给 <项目> 新增一种 <主题中的某种变体>,叫 <具体名字>,
>  需要改哪几个文件?分别改什么?"

如果你的答案是:
1. <文件 A>:<改什么>
2. <文件 B>:<改什么>
3. ...

**注意:你必须 / 不必修改 <主角文件>**。<点出关键>

对比一下 <对照案例> 的同一道题:<差异>。

这就是<本案例模式名>与<对照案例模式名>在 <某原则>上的真实差距。
```

**关键**:**镜像对应对照案例里的同一道题**。两个答案的差异往往就是整篇案例的论点。例如 Redis 那道题答案是"必须改 evict.c",Linux TCP 同样的题答案是"不必改任何客户端"——这对差异本身就是博客的核心论点。

## 写作时的几个易踩坑点

- **不要按文件目录顺序介绍文件**。按"从抽象到具体"的认知路径,顺序应该跨文件交错(可能先读 server.h 的一段,再读 config.c,再回到 server.h 的另一段)。
- **不要在指南里大段贴代码**。指南是引路,不是教程。贴代码不超过 5~10 行,且必须配合"看的时候注意 X / 读完应该明白 Y"的引导。
- **不要漏掉差异表**。这是案例的灵魂——没有这张表,案例只是"看懂了 Redis 怎么写",有了它案例才变成"看懂了一种模式的多种长相"。
- **不要把"还想深入的话"写成总结**。它是给少数想看完整版的人留的指路,不是案例的结论。结论应该在"建议练习"那一节自然落定。
