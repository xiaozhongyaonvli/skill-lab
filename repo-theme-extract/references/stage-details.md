这份文档是 SKILL.md 的展开。每次开始一个阶段前,打开对应那一节。

## § 阶段 1 · 调研仓库结构

### 命令模板

```bash
# 列某个目录下的文件清单
gh api repos/<owner>/<repo>/contents/<path>

# 例:列 redis 仓库的 src/ 目录
gh api repos/redis/redis/contents/src

# 不知道目录结构时,先列根目录
gh api repos/<owner>/<repo>/contents
```

返回是 JSON,可以管道到 `jq -r '.[].name'` 只看文件名。

### 入口反推法的详细步骤

**Step A · 从主题反推一个候选入口名**

很多情况靠常识就能猜:
- "内存淘汰" → `evict.c` / `eviction.c`
- "查询规划" → `planner.c` / `planner/` 目录
- "raft 共识" → `raft.go` / `raft/` 目录
- "TCP 拥塞" → `tcp_cong.c` / `tcp_congestion.c`

猜不准时:`gh api .../contents/src | jq -r '.[].name'`,**只看文件名**挑出 1~2 个最像的候选。

**Step B · 下载入口文件,grep 三类符号**

下载方法见阶段 2,但这里**先临时下一份用来看**就行,不要急着按命名约定保存。

在入口里 grep:

| 符号类型 | 例(Redis 内存淘汰) | 指向的关联文件 |
|---|---|---|
| 类型/字段引用 | `server.maxmemory_policy`、`MAXMEMORY_FLAG_*` | `server.h`(宏定义和字段定义所在) |
| 注册/配置入口 | `maxmemory_policy_enum` | `config.c`(配置项注册表) |
| 数据结构副作用 | `->lru` 字段在多处使用 | `object.c`(创建对象路径里有它的初始化) |

对每个 grep 出来的符号,用 `gh api .../contents` 或者 GitHub 的代码搜索(`gh api search/code?q=...`)定位它在哪个文件里定义/注册。

**Step C · 判断清单是否够全**

四角色对照表:

| 角色 | 这个文件回答什么 |
|---|---|
| (a) 主题身份 | 主题相关的常量、宏、状态字段定义在哪 |
| (b) 主题选择/注册 | 用户怎么挑选/启用某种实现(配置项注册、命令解析) |
| (c) 主题使用(主角) | 主题逻辑实际跑在哪里(核心分发函数) |
| (d) 主题对存储/状态的反向影响 | 主题决策会修改哪些数据(对象元信息、统计字段) |

**如果 4 个角色里有空缺或有重复,清单还没准备好**——再 grep 一轮。

### 易踩坑点

- **不要靠目录树名字硬猜**。某些项目用模块化布局(`pkg/eviction/`),某些用平铺(`src/evict.c`)。统一用 `gh api ... | jq -r '.[].name'`。
- **不要漏掉头文件**。C/C++ 项目里宏定义和结构体声明往往在 `.h`,grep 完 `.c` 一定要再追到 `.h`。
- **不要把"测试文件"误当成关联文件**。`evict.c` 关联 `server.h`,不关联 `test_evict.c`。

## § 阶段 2 · 部分下载

### 命令模板

```bash
# 下载单个文件,保留原仓库子路径
mkdir -p <案例名>/<repo-name>/src
curl -sL https://raw.githubusercontent.com/<owner>/<repo>/<ref>/<path> \
  -o <案例名>/<repo-name>/<path>

# 例
curl -sL https://raw.githubusercontent.com/redis/redis/unstable/src/evict.c \
  -o redis内存淘汰策略/redis/src/evict.c
```

### ref 怎么选

优先级:
1. **release tag**(例 `7.4.0`):稳定,版本号可查,日后回看不会因仓库改动而对不上
2. **default branch**(例 `unstable` / `main`):图方便时可以,但要在阅读指南里记录"基于 commit XXX 时的状态"
3. **从不**用某个 commit hash,除非用户明确指定——commit hash 对读者不友好

**问用户**:如果用户没指定 ref,默认选 release tag,告诉他们选了哪个。

### 命名约定(关键)

| 文件类型 | 命名 | 处理 |
|---|---|---|
| 主角文件(整篇都是主题相关) | 直接原名(例 `evict.c`) | 在原地精简、原地翻译 |
| 巨型辅助文件(大部分跟主题无关) | `<原名>.full`(例 `server.h.full`) | 用作精简对照,精简后另存为原名,完成后 `rm *.full` |

主角文件 = 通常是阶段 1 找到的那个入口文件。其它都按巨型辅助文件处理。

**为什么这么做**:精简时打开原版 + 精简版做左右对照,做完直接 `rm`,不污染最终目录。这条约定在阶段 3 立刻见效。

### 易踩坑点

- **不要拉整个目录**。`gh repo clone` 是禁止动作。
- **保留子路径**。`redis/src/evict.c` 而不是 `evict.c` 平铺。
- **大文件检查**:如果 curl 拉下来的文件 > 5000 行,优先怀疑是不是选错了文件——一般来说真正"密切相关"的文件不会那么大,可能是选了整个 `server.c` 而该选 `evict.c`。

## § 阶段 3 · 精简代码

### 三类代码的去留(细则)

**(a) 主题核心代码 · 一行不删 + 加学习注释**

判断标准:这段代码"对主题来说是必须的",拿掉它整个主题逻辑就不成立。例:
- 分发结构(`if/else` 链、函数指针表、switch)
- 数据结构里直接服务主题的字段
- 主题的对外接口(命令、配置项、API)

学习注释怎么写:用 `/* ★ ... */` 或 `/* ★★★ ... */` 标记,**说明"为什么这段是核心"**而不是"这段做什么"——后者代码已经说了。

例:
```c
/* ★★★ 这里没用函数指针表,而是用 if/else 大分发。理由是 Redis 把
 *      策略选择放到了客户端(performEvictions),让具体存储(robj)只
 *      记录原始指标(lru/lfu),不知道"自己被哪种策略管理"。这是和
 *      Linux TCP 拥塞控制最大的差异点——后者用 struct
 *      tcp_congestion_ops 把策略变成可注册的对象。 */
if (server.maxmemory_policy & MAXMEMORY_FLAG_LRU) {
    ...
} else if (server.maxmemory_policy & MAXMEMORY_FLAG_LFU) {
    ...
}
```

**(b) 主题辅助代码 · 保留 + 翻译注释**

判断标准:被核心代码调用,但本身不是模式重点(工具函数、定时器、内存计量)。

只翻译原注释,不加 `★`。例:
```c
/* 估算对象的空闲时间(基于 server.lruclock - obj->lru)。
 * 注意 lruclock 是个截断到 24bit 的时间戳,会回绕。 */
unsigned long long estimateObjectIdleTime(robj *o) { ... }
```

**(c) 与主题正交的代码 · 删 + 占位说明**

判断标准:跟主题无关的命令实现、AOF/复制/集群/ACL/TLS/Lua/模块、其它配置项。

占位写法:
```c
/* ... (省略约 400 行:命令表、AOF、复制、集群、ACL、TLS、Lua、模块
 *      相关字段和函数) ... */
```

**说明省略了什么**,这样读者知道"这里原本有东西,只是不相关",不会以为是漏抓了。

### 判断"删够了"的硬标准

精简完每个文件,**用一句话**写出它在主题里扮演什么角色。说不出来 = 还没精简够,继续删。

这一句话最终会进入阅读指南"文件角色对照表"。

### 易踩坑点

- **不要把整个结构体保留**。结构体几百个字段里通常只有 5~10 个跟主题相关。把无关字段全删掉,用占位说明。
- **不要把所有命令实现保留**。命令实现往往最长,而且 90% 跟主题无关。
- **保留子路径**。删完后还在 `redis/src/server.h`,不要挪到根目录。

## § 阶段 4 · 翻译 + 学习注释

### 文件头"学习注释 / 精简说明"块的三段式

每个文件开头都写一段,严格三段:

```
/* ============================================================
 * 学习注释 / 精简说明
 *
 * 【原文件】<相对路径> · <原行数>行 · 负责<一句话职责>
 *
 * 【本主题保留】
 *   - <段落 1>(原 line A~B)
 *   - <段落 2>(原 line C~D)
 *   - ...
 *
 * 【本主题删除】
 *   <一句话概括删了什么大类>
 *
 * 【对比锚点】
 *   <把这个文件跟某个教科书概念 / 对照案例某文件挂钩,一句话>
 * ============================================================ */
```

例(Redis 案例 server.h):
```
【原文件】src/server.h · 4622行 · Redis 全局头文件,定义服务器主结构与所有宏

【本主题保留】
  - MAXMEMORY_FLAG_* / MAXMEMORY_* 全部 10 个宏(原 line 685~701)
  - redisServer 中 6 个 maxmemory* / lfu_* 字段(原 line 2417~2425)
  - configEnum 类型定义(原 line 3928~3931)
  - LFU_INIT_VAL 与 EVICT_* 常量(原 line 4214~4220)
  - evict.c 的对外函数声明 6 个

【本主题删除】
  所有命令表、AOF、复制、集群、ACL、TLS、Lua、模块相关字段和声明

【对比锚点】
  对应 Linux TCP 案例的 include/net/tcp.h——都是"把策略相关的常量
  和函数声明集中放置"的那个总头文件。
```

### 翻译要"再加工",不是机翻

具体做法:
- 把 4~6 行的啰嗦英文段落压成 2~3 行紧凑中文
- 保留意图,去掉句法冗余("It should be noted that..." → 不译)
- 中文里能用顿号、破折号替代连词,就替代

例:
- 原文:`/* Note that we should also try to do this in a way that is friendly to the CPU cache, because random reads from a big hash table can be slow. */`
- 机翻:`/* 注意我们也应该尝试以对 CPU 缓存友好的方式做这件事,因为从大哈希表随机读可能很慢。 */`
- 再加工:`/* 也要照顾 CPU 缓存——大哈希表的随机读很慢。 */`

### `★` / `★★★` 标注规则

- `★`:本文件的次要决策点(2~3 处)
- `★★★`:本文件的主要决策点,通常只有 1 处(主角文件可能 2~3 处)

整个产物加起来 `★★★` 不要超过 6 个。**过多则失效**——读者扫读时看到一片 ★ 就会失去焦点。
