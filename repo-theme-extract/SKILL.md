# repo-theme-extract

## 这个 skill 在做什么

把一个庞大的开源仓库,**只就一个主题**做"精读样本"。最终产物不是一份博客或讲义,而是一组**可读、可对照、可练习**的精简代码 + 一份阅读指南。读者拿到这堆东西,能在 1~2 小时内看懂这个主题在该项目里的实现脉络。

为什么需要这个 skill:大项目的代码动辄上万行,直接 clone 下来读没人读得动;但只要把"和主题正交的代码"全部砍掉、原英文注释翻成压缩过的中文、关键决策点用 `★` 标出来,再配一张"建议阅读顺序"的图,这同一份代码立刻可读性翻倍。

## 输入 / 输出

**输入两样东西**(如果用户没给齐,要主动问):
1. 一个开源仓库 URL,例如 `https://github.com/redis/redis`
2. 感兴趣的主题,例如"内存淘汰策略中关于策略模式的部分"

**输出**:
```
<工作目录>/<案例名>/<repo-name>/<原仓库子路径>/<精简后文件>
<工作目录>/<案例名>/阅读指南.md
```

例:
```
笔记/第一章/项目/redis内存淘汰策略/
├── 阅读指南.md
└── redis/src/
    ├── evict.c
    ├── server.h
    ├── config.c
    └── object.c
```

**两条硬约束**:
- **绝不全量克隆仓库**。只下载选中的文件。
- **只用两个外部工具**:`gh api`(列目录、查元信息)和 `curl https://raw.githubusercontent.com/...`(下载文件原文)。

## 工作流(五阶段)

每个阶段下面只列**主干**和**最关键的一两条判断**。完整的判断标准、命令模板和反例,见 `references/stage-details.md`——开始做任何一个阶段前,请先打开对应那一节。

### 阶段 1 · 调研仓库结构(只列清单,不下载)

用 `gh api repos/<owner>/<repo>/contents/<path>` 列目录,从主题反推**入口文件名**(例:"内存淘汰" → `evict.c`,这一步常常靠常识)。然后**在入口文件里 grep 三类符号**:
- 类型/字段引用(如 `server.maxmemory_policy`)
- 注册/配置入口(如 `xxx_policy_enum`)
- 数据结构副作用(如 `->lru` 字段)

每类符号都指向一个关联文件。**判断"清单是否够全"的标准**:这堆文件应该正好覆盖"主题身份 / 主题选择 / 主题使用 / 主题对存储的反向影响"这四个角色——不重复、不遗漏。

详情和反例:`references/stage-details.md` § 阶段 1。

### 阶段 2 · 部分下载

`curl -sL https://raw.githubusercontent.com/<owner>/<repo>/<ref>/<path>` 拉文件。**ref 优先用 release tag,其次默认分支**;如果用户没指定,问一下。

**命名约定**(关键):
- 整篇都是主题相关的文件(通常是"主角文件",例 `evict.c`)→ **直接用原名**保存,在原地精简。
- 大部分跟主题无关的"巨型文件"(几千行的头文件、配置文件)→ 用 **`.full` 后缀**(例 `server.h.full`)保存原始版,精简后另存为原名(`server.h`),精简完成后 `rm *.full`。

这个约定的好处在阶段 3 立刻体现:精简时左右对照,做完直接 `rm`,不污染最终目录。

详情:`references/stage-details.md` § 阶段 2。

### 阶段 3 · 精简代码

每一行代码归到三类之一:
- **(a) 主题核心代码**——一行都不删,**加学习注释**解释"为什么这里是核心"。
- **(b) 主题辅助代码**——保留,但只翻译/简化原注释,不再额外加学习注释。
- **(c) 与主题正交的代码**——删掉,用 `/* ... (省略 ...) ... */` 占位并说明省略了什么。

**判断"删够了"的标准**:每个文件读完应该能用**一句话**说明它在主题里扮演什么角色(后面阅读指南里"文件角色对照表"就是这一句话的集合)。说不出来 = 还没精简够。

**保留原仓库子路径**(例:`redis/src/` 而不是平铺到根目录),让读者对照原仓库时心智成本最低。

详情:`references/stage-details.md` § 阶段 3。

### 阶段 4 · 翻译 + 学习注释

**每个文件头加一块"学习注释 / 精简说明"**,三段式:
1. 原文件多大、负责什么;
2. 本主题保留了哪几段、删除了什么;
3. 一个【对比锚点】把这个文件跟教科书或同类对照案例挂钩(例 Redis 案例全程对照 Linux TCP 拥塞控制)。

**翻译要"再加工"而不是机翻**:原文 4~6 行的啰嗦英文段落压成 2~3 行紧凑中文,保留意图,去掉句法冗余。这一步明显减少最终文件长度,也让节奏更接近读者扫读速度。

**`★` 标注主题命中点**:每个文件 1~3 处 `★` 或 `★★★`,扫读时一眼能找到决策点。不要滥用——满屏都是 ★ 等于没有 ★。

详情:`references/stage-details.md` § 阶段 4。

### 阶段 5 · 写阅读指南.md

这是产物里**质量最重要的一份文件**——它告诉读者"按什么顺序读、每个文件什么角色"。完整结构、模板和示例见 `references/reading-guide.md`。骨架:

1. ASCII 大图,把所有文件按层级摆好(用户态 → 注册表 → 全局状态 → 客户端 / 具体逻辑)。
2. **必备两张表**:
   - "文件角色对照表":每行一个文件,列出"角色 / 行数 / 一句话职责"。
   - "和对照案例的差异表":把本案例和一个**已经写过的对照案例**(如 Linux TCP)按 5~10 个维度并排比较——这是把案例从"读懂代码"提升到"看懂模式"的关键。
3. **建议阅读顺序**:不是按文件目录顺序,而是按"从抽象到具体"的认知路径(身份 → 注册 → 主角 → 细节 → 副作用)。
4. **结尾留一道练习题**(约 200 字):"如果要新增一种 X,需要改哪些文件?分别改什么?"——答得出来就说明读进去了。

## 五条核心规则(这是 skill 的灵魂,别忽略)

这五条是从一次完整案例(`references/case-redis.md`)里反推出来的工程性规则,下次做新案例时直接遵循即可:

1. **入口反推法**:主题 → 入口文件名(常识 or 列目录)→ 在入口里 grep 三类符号(类型/字段、注册/配置、数据结构副作用)→ 得到关联文件清单。**先列清单,再下载**——而不是边下边看。
2. **`.full` 后缀约定**:精简过程中保留原始下载件供左右对照,精简完 `rm`。**主角文件直接原名保存**。
3. **★ / ★★★ 标注主题命中点**:每个文件 1~3 处即可,过多则失效。
4. **必备产出表格两张**:文件角色对照表 + 和对照案例的差异表(后者尤其重要——它承担"从读懂代码到看懂模式"的跨越)。
5. **结尾留一道"加一个新 X 要改哪里"的练习题**,镜像对应对照案例里的同一道题——两个答案的差异往往就是整篇案例的论点。

## 一个常见的失败模式提醒

新手最容易犯的错:**直接 `git clone` 整个仓库,然后开 IDE 一通乱翻**。这违反"不全量克隆"和"先列清单"两条约束,而且会让你陷入"看到啥都觉得相关"的认知陷阱。本 skill 的整条工作流就是为了避开这个失败模式——**坚持先列清单(阶段 1)、再下载(阶段 2)、再精简(阶段 3)**,不要倒序也不要跳步。
