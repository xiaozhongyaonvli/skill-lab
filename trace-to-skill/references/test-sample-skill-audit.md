# Skill 审查测试样例：规划合理性与跳步风险检查

## 目的

用这个样例测试 `trace-to-skill` 是否能把一次真实的 skill 审查流程记录下来，并在结束后自动抽象成一个可复用的审查型 skill 草稿。

这个测试样例是自包含的，不要求再去读取额外文档。测试时，直接按下面给出的审查步骤和对话脚本执行即可。

## 适合测试的真实场景

用户给 agent 一个现成 skill 仓库或一个 `SKILL.md`，要求 agent 审查这个 skill 的规划是否真正约束执行，而不是只写了几个步骤。

这次审查重点不是“有没有 plan”，而是要判断下面这些事：

1. 是否有唯一入口和清晰主链路
2. 规划是否外部化成正式工件
3. 执行是否必须继承规划
4. 是否存在跳步、跑偏或第二套真相面
5. 完成声明是否绑定验证和 cleanup

## 这次测试里要检查清楚的 SOP

在实际录制里，agent 应该按这个顺序推进，不要跳着看。

### Step 1：先锁唯一入口和主执行链

先看：

- `SKILL.md`
- `runtime.md`
- `workflow.md`
- `instruction.md`
- 主调度脚本

要回答：

1. 用户从哪个入口进入这个 skill
2. 内部有没有固定阶段顺序
3. 执行是不是必须沿着主链推进

如果主链路本身不清楚，不要直接下总体结论。

### Step 2：检查规划是否外部化成正式工件

重点看是否存在这些产物：

- requirement
- plan
- progress
- findings
- contract
- receipt
- manifest

要判断：

1. 规划是写在文件里，还是只存在聊天里
2. 需求、计划、进度、证据是否分层存放
3. 各阶段是否有对应产物
4. 后续执行是否会引用这些产物

### Step 3：检查规划内容本身是否完整

至少看有没有这些要素：

- 目标
- 阶段划分
- 每阶段完成条件
- 关键问题
- 决策与理由
- 错误记录
- 验证方式
- 回滚方案
- cleanup 预期
- ownership boundaries

要判断它写的是“执行契约”，还是只是一份步骤清单。

### Step 4：检查执行是否必须继承规划

重点排查：

- requirement / plan 有没有冻结
- 执行前会不会回读当前计划
- 能不能从思考直接跳到 ad-hoc 执行
- 有没有固定的先后顺序约束

如果 execution 可以绕开规划直接做事，那这个 skill 的规划约束就是假的。

### Step 5：检查是否存在第二套真相面

重点排查：

1. 谁可以定义 canonical requirement
2. 谁可以定义 canonical plan
3. 谁可以宣布完成
4. 子 agent 或 specialist 能不能自己重写规划

如果多个执行单元都能改目标、改计划、宣布完成，这个 skill 就容易失去唯一真相面。

### Step 6：检查是否有持续参与执行的防跳步机制

重点看：

- pre-tool hook
- post-tool hook
- stop hook
- session catchup
- findings / progress 更新规则

要判断：

1. 计划会不会被持续回读
2. 调研证据会不会被外部化
3. 阶段状态会不会持续同步
4. 中断恢复后能不能找回当前阶段
5. 收尾前会不会检查未完成项

### Step 7：检查是否存在静默降级和代理目标漂移

重点排查：

1. 主路径失败时是否允许 silent fallback
2. fallback 后是否会明确说明降级
3. 会不会把局部成功冒充总体完成
4. completion state 是否区分 complete / partial / blocked

### Step 8：检查完成声明是否绑定验证和 cleanup

要回答：

1. 完成前是否必须有验证证据
2. cleanup 是正式阶段还是随手处理
3. 是否有完成收据或可追踪结果
4. 如果没有验证或没有 cleanup，系统还能不能直接宣布完成

## 建议录制开始命令

```text
@trace-start skill-audit-plan-governance
```

## 推荐测试对话脚本

下面的内容建议按多轮分别发送，不要一次性贴完。

### Turn 1：声明目标

```text
我要你帮我审查一个 skill，重点不是看它有没有写计划，而是看它的规划是否真的能约束执行，是否存在跳步风险。
```

### Turn 2：限定检查对象

```text
审查对象先从这个 skill 的主入口开始：
- `SKILL.md`
- 如果有 runtime、workflow、instruction 或主调度脚本，也一起纳入
```

### Turn 3：要求先锁主链路

```text
先回答这几个问题：
1. 用户从哪个唯一入口进入
2. 有没有固定阶段顺序
3. 执行是不是必须沿着主链推进

如果主链不清楚，不要直接下总体结论。
```

### Turn 4：要求检查规划外部化

```text
然后检查规划有没有外部化成正式工件。
我想看到 requirement、plan、progress、findings、contract、receipt 这类东西是否存在，是否分层存放，后续执行是否引用它们。
```

### Turn 5：要求检查规划完整性

```text
再检查规划内容本身是否完整。
重点看是否有目标、阶段划分、阶段完成条件、决策理由、错误记录、验证方式、回滚方案和 cleanup 预期。
```

### Turn 6：要求检查执行是否继承规划

```text
再检查 execution 能不能绕开规划直接做事。
重点看有没有冻结 requirement / plan、执行前是否回读计划、是否允许从思考直接跳到 ad-hoc 执行。
```

### Turn 7：要求检查第二套真相面和防跳步机制

```text
继续检查：
1. 有没有第二套真相面
2. 子 agent 能不能重写 canonical requirement 或 canonical plan
3. 有没有防跳步、防跑偏、防静默降级的机制
```

### Turn 8：要求检查完成声明

```text
最后检查完成声明是不是绑定验证和 cleanup。
如果没有验证证据、没有 cleanup 产物，就不要算真正完成。
```

### Turn 9：要求固定结论格式

```text
结论请固定按这 5 个维度输出：
1. 规划外部化是否充分
2. 执行继承规划是否严格
3. 跳步风险是否可控
4. 真相面和权限面是否清晰
5. 完成声明是否有证据支撑
```

### Turn 10：要求产出结构化审查结果

```text
请给我一个结构化审查结果，包含：
- 总体判断
- 主要优点
- 主要风险
- 改进建议
```

## 建议录制结束命令

```text
@trace-stop
```

## 这次录制理应抽象出的 SOP

如果 `trace-to-skill` 表现正常，生成的草稿应该能恢复出类似这样的稳定流程：

1. 锁定唯一入口和主执行链
2. 检查规划是否外部化成正式工件
3. 检查规划内容是否完整且可审查
4. 检查执行是否必须继承规划
5. 检查是否存在第二套真相面
6. 检查是否有持续参与执行的防跳步机制
7. 检查是否存在静默降级和代理目标漂移
8. 检查完成声明是否绑定验证与 cleanup
9. 用固定结论模板输出审查结果

## 录制结束后应重点检查什么

检查生成结果是否真正抽象出了“审查型 SOP”，而不是只复述了对话内容。

重点看：

- 是否保留了固定检查顺序
- 是否保留了“先主链、后工件、再执行、再治理、最后验证”的阶段结构
- 是否把检查问题抽象成可复用 decision rules
- 是否把“如果主链不清楚，不要下结论”这类约束保留下来
- 是否把“验证和 cleanup 缺失时不能宣布完成”保留下来

## 通过标准

这个测试样例算成功，至少要满足下面几点：

- 生成草稿能识别出这是一个“skill 审查”类场景
- 生成草稿能提炼出固定阶段顺序，而不是只抄录聊天
- 生成草稿能把关键检查点抽象成 SOP 或 decision rules
- 生成草稿能区分“检查对象”“检查顺序”“结论格式”“完成条件”
- 生成草稿里的 `SKILL.md` 草案看起来像一个能复用的审查 skill，而不是一次性总结
